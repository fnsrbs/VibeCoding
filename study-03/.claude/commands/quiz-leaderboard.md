src/store/quizStore.ts와 src/types/quiz.ts 파일을 읽고 순위 시스템을 분석하고 관리해줘.

**실행 모드:**
`$ARGUMENTS`가 비어 있으면 → 순위 시스템 전체 분석 리포트 출력
`$ARGUMENTS`가 `"audit"` → 랭킹 로직 상세 감사 (엣지 케이스 포함)
`$ARGUMENTS`가 `"simulate"` → 주요 플레이 시나리오별 예상 점수 시뮬레이션
`$ARGUMENTS`가 `"improve"` → 현재 시스템의 문제점과 개선안 제안

## 1. 랭킹 데이터 구조 분석

`RankingEntry` 타입과 `useQuizStore`의 `saveRanking`, `rankings` 필드를 읽고 아래를 정리해줘.

| 필드 | 타입 | 역할 |
|------|------|------|
| nickname | string | ... |
| score | number | ... |
| category | CategoryFilter | ... |
| date | string | ... |
| savedAt | number | ... |

- 저장 한도: 최대 몇 개까지 보관되는지
- 정렬 기준: 어떤 기준으로 순위가 결정되는지
- 동점 처리: 동점자 간 순서가 어떻게 결정되는지 (코드 기준)
- persist 범위: localStorage에 저장되는 필드와 세션 중에만 존재하는 필드 구분

## 2. 점수 획득 경로 분석

`calcPoints` 함수와 `finishQuiz` 로직을 기반으로 점수 흐름을 정리해줘.

```
정답 여부 → 기본 점수 → 스피드 보너스 → 콤보 보너스 → 만점 보너스
```

각 단계에서 조건과 획득 점수를 명시하고, **최고 점수를 내기 위한 최적 전략**을 한 문단으로 설명해줘.

## 3. 이론적 최고 점수 계산

아래 두 모드의 이론적 최고 점수(퍼펙트 플레이 기준)를 계산하고 근거를 보여줘.

- **카테고리 모드** (10문제): 조건별 획득 점수 합산 + 만점 보너스
- **전체 도전** (40문제): 조건별 획득 점수 합산 + 만점 보너스

## 4. 엣지 케이스 / 랭킹 공정성 검토

아래 항목을 코드 기반으로 검토하고 이슈가 있으면 ⚠️ 표시해줘.

1. **동점 처리**: 같은 점수일 때 먼저 등록된 기록과 나중 기록의 순서가 일관되게 처리되는지
2. **중복 등록**: 같은 닉네임으로 여러 번 등록할 수 있는지, 이로 인한 순위 왜곡 가능성
3. **카테고리 필터**: 랭킹 페이지의 탭 필터(`all` 탭)가 카테고리 모드 점수와 전체 도전 점수를 함께 집계할 때 공정한지
4. **만료 없는 랭킹**: 랭킹에 만료 기간이 없어 초기 고득점자가 영구적으로 상위권을 점유할 가능성
5. **localStorage 용량**: 최대 100개 저장 시 실제 예상 용량(KB) 계산

## 5. 개선 제안

현재 랭킹 시스템에서 아래 측면별로 개선 아이디어를 각 2~3줄로 제안해줘.

- **공정성**: 카테고리 간 최고 점수 차이를 줄이는 방법
- **경쟁성**: 플레이어들이 더 자주 도전하게 만드는 순위 메커니즘
- **투명성**: 점수 계산 과정을 UI에서 더 잘 표시하는 방법
- **데이터**: 현재 `RankingEntry`에 추가하면 유용한 필드 (정답률, 소요 시간 등)
