## 단계 3/7: 카테고리별 부족 분석

단계 2 결과를 기준으로 각 카테고리의 현황을 진단해줘.

판단 기준:
- **목표 문제 수**: 카테고리당 10문제 (현재 앱 로직 기준)
- **목표 난이도 비율**: easy 30% / medium 40% / hard 30% (±10%p 허용)
- **정답 인덱스 분포**: 0~3번 각각 20~30% 범위가 이상적 (40% 이상 집중 시 ⚠️)

카테고리별 출력 형식:
```
[카테고리명]
  문제 수: {현재} / 목표 10개 → {부족 or 충족 or 초과}
  난이도: easy {n}% / medium {n}% / hard {n}% → {이상 있으면 ⚠️ 항목 명시}
  정답 인덱스: 0:{n}% / 1:{n}% / 2:{n}% / 3:{n}% → {편향 있으면 ⚠️}
  추가 권장: {필요한 난이도와 수량, 없으면 "없음"}
```

마지막에 전체 카테고리의 우선 보완 순위를 1~4위로 요약해줘.

## 단계 4/7: 새 문제 추가 전 중복 체크

`$ARGUMENTS`에 추가할 문제 JSON이 제공된 경우: 해당 문제들을 대상으로 아래 검사 수행.
`$ARGUMENTS`가 비어 있는 경우: 이 단계를 건너뛰고 "⏭ 단계 4 건너뜀 — 추가할 문제 없음" 출력 후 단계 5로 이동.

중복 검사 항목:
1. **ID 중복**: 제공된 문제의 `id`가 기존 데이터에 이미 존재하는지
2. **문제 텍스트 유사도**: `question_text`가 기존 문제와 90% 이상 유사한 것이 있는지 (의미상 동일 여부 판단)
3. **타입 필드 검증**: `id`, `question_text`, `options`(4개), `correct_index`(0~3), `explanation`, `category`, `difficulty` 필드가 모두 존재하고 올바른 타입인지
4. **카테고리 값 검증**: `category`가 `한국사`, `과학`, `지리`, `일반상식` 중 하나인지
5. **난이도 값 검증**: `difficulty`가 `easy`, `medium`, `hard` 중 하나인지

검사 결과:
- 모든 검사 통과 시 → "✅ 중복 없음, 형식 정상 — 추가 가능" 출력
- 하나라도 실패 시 → **즉시 중단**, 실패한 항목과 문제 ID를 명시

## 단계 5/7: 문제 추가 후 형식 검증

`$ARGUMENTS`에 문제가 있어서 단계 4를 실제로 수행했고 통과한 경우에만 이 단계 실행.
그 외(단계 4를 건너뛴 경우)에는 "⏭ 단계 5 건너뜀" 출력 후 단계 6으로 이동.

`src/data/questions.ts` 파일의 `questions` 배열 끝에 단계 4를 통과한 문제를 추가해줘.

추가 후 즉시 아래 검증을 수행해줘:
1. 파일이 유효한 TypeScript 문법인지 (`tsc --noEmit` 실행)
2. 추가된 문제가 배열에 실제로 존재하는지
3. 전체 문제 수가 (이전 수 + 추가한 수)와 일치하는지
4. 추가된 문제의 ID가 전체 ID 목록에서 중복 없이 유일한지

검증 실패 시 → 추가한 내용을 즉시 원래대로 되돌리고 **즉시 중단**.
